## The WHY

Why does today's concept exist? What broke without it?

Every data structure has a cost. Choosing the wrong one doesn't just make code slow….. it makes it slow in a way that gets worse the more your system grows. Today was about proving those costs with real numbers and learning to make tradeoffs consciously

## Key insight

The thing that surprised you or clicked today — in your own words.

Faster searching costs memory. Systems engineering is a tradeoff between speed and cost. There is no free lunch — a set gives you instant lookups but costs 8x more memory than a tuple. The job of the engineer is to know the tradeoffs and choose deliberately, not out of habit

## What I built

Project progress, code written, experiments run.

- `containers.py` — created list, tuple, and set with the same data. Measured memory with `sys.getsizeof()`. Tuple: 88 bytes. List: 120 bytes. Set: 728 bytes. Same 5 items, wildly different costs.
- `lookup_timing.py` — timed list vs set lookup at 100K items. Set was 1,438x faster.
- `lookup_scaling.py` — timed lookups at 1K, 10K, and 100K. List time grew 10x with each 10x increase in data (O(n)). Set time stayed flat (O(1)).
- `lookup_plot.py` — first matplotlib chart. Plotted both curves. The visual made O(n) vs O(1) impossible to forget.

Same principle as the Python lesson — separate cheap work from expensive work.
